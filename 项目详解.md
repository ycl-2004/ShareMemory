# ShareMemory 项目详解(内部文档)

> 这份文档是给项目作者自己看的完整设计说明,从问题、思路、架构到每一条规则的"为什么"。公开的简介看 `README.md`,给 agent 执行的规范看 `MEMORY_PROTOCOL.md`,本文不参与 agent 的启动加载(不耗 token)。

---

## 一、要解决的问题

Claude Code 和 Codex 是两个独立的 AI 编程 agent,它们之间存在三个根本断裂:

1. **互相不可见**:Claude 做的架构决策、装的依赖、踩过的坑,Codex 完全不知道,反过来也一样。两个 agent 在同一个项目里很容易做出矛盾的决定(一个选了 Next.js,另一个按 Vite 来写)。
2. **跨 session 失忆**:即使只用一个 agent,每次新开 session 它也是"空白脑"进场,要么重新读一遍整个代码库(贵),要么瞎猜(危险)。
3. **没有共同的事实源**:口头(prompt)传达项目约定不可靠,会随着对话变长被遗忘。

平台层面没有解法——两家产品不共享任何内部状态,以后也不太可能共享。

## 二、核心思路:不做共享内存,做共享事实源

真正的"共享内存"做不到,但可以做**伪共享记忆层(fake shared memory layer)**:

- 在项目里放一组 Markdown 文件(`AI_MEMORY/`)作为唯一事实源(single source of truth);
- 利用两个产品的共同点——**都会自动加载项目根目录的指令文件**(Claude Code 读 `CLAUDE.md`,Codex 读 `AGENTS.md`)——强制它们启动时读这组文件、工作中写这组文件;
- 两个 agent 在同一台机器同一个文件夹工作,读写的是物理上同一份文件,于是"我写你读"就等价于共享了记忆。

记忆分为两个心智层:

- **交接历史层**:`SYNC_LOG.md` + `archive/`,按天记录 handoff 摘要,一天最多一个日期块。它回答"最近发生了什么"。
- **当前视图层**:`PROJECT.md` / `DECISIONS.md` / `TASKS.md` / `LEARNINGS.md`,保存当前仍有用的事实。它回答"现在应该按什么前提继续做"。

三个刻意的设计取舍:

- **只做项目级,不做全局**。Codex 和 Claude Code 的规则文件本来就是 project-based,全局偏好放各自的 `~/.claude/CLAUDE.md` / `~/.codex/AGENTS.md` 即可,不属于这个框架。范围越小,记忆越不容易膨胀。
- **不依赖任何平台专属功能**。同步全靠文件和规则约定,所以未来加 Cursor、加任何新 agent,只要给它一份指向 `MEMORY_PROTOCOL.md` 的启动文件就能加入。
- **不用 git 做同步,只用 git 做找回**。同一文件夹不存在"传输"问题;git 的价值是历史版本——内容被覆盖时能恢复(详见第八节)。

## 三、整体架构

```
分发层   本仓库 = share-memory skill(SKILL.md 在根目录)
         安装:~/.claude/skills/(Claude Code) + ~/.agents/skills/(Codex)
         init 时把下面两层复制进目标项目
                              │
启动层   CLAUDE.md(Claude 读)      AGENTS.md(Codex 读)
              │                          │
              └───────► MEMORY_PROTOCOL.md ◄───────  (共同规范,唯一规则源)
                              │
记忆层                   AI_MEMORY/
              CONFIG · PROJECT · DECISIONS · TASKS
              LEARNINGS · SYNC_LOG · archive/
                              │
工具层   scripts/check_memory.sh (写后校验) · git (可选,历史找回层)
```

**分发模型:为什么仓库本身就是 skill**。最初的设计是"项目模板"模式——把整个文件夹复制进项目,SKILL.md 藏在 `.claude/skills/share-memory/` 里(Claude Code 自动发现项目级 skill 的位置)。后来改为"仓库 = skill"模式,因为 Codex 自 2025 年 12 月起也支持 skills(`~/.agents/skills/` 下放 SKILL.md,frontmatter 格式与 Claude Code 相同)。现在同一个文件夹两边都能安装:装一次,所有项目都能用 "init memory" 一键铺设;而项目里真正常驻的只有 init 复制进去的启动文件和协议,skill 本体按需加载,不占项目空间。已有 CLAUDE.md/AGENTS.md 的项目,init 只插入或替换 SHAREMEMORY marker block,改前备份,不碰用户内容。

**为什么协议单独成一个文件**:CLAUDE.md 和 AGENTS.md 如果各写一遍完整规则,改一条要改两处,迟早不一致。所以两个启动文件都只有 ~15 行"速查表 + 你必须读协议",真正的规则只存在于 `MEMORY_PROTOCOL.md` 一处(DRY 原则)。

**为什么启动文件要保持极小**:CLAUDE.md 这类文件会进入每个 session 的常驻上下文,每一行都在每轮对话烧 token。业界共识是常驻指令文件要像"查找表"而不是"知识库"。协议文件每个 session 只读一次,成本可控。

## 四、AI_MEMORY/ 每个文件的职责

| 文件 | 内容 | 谁写 | 何时写 |
|---|---|---|---|
| `CONFIG.md` | 记忆语言、git 开关、创建日期、协议版本 | init 时一次 | 基本不再改 |
| `PROJECT.md` | 项目概览、架构、约定 + **Long-Term Memory 区** | 两个 agent | 项目目标/结构变化时自动写;Long-Term 区由溢出蒸馏而来 |
| `DECISIONS.md` | 架构/技术决策、依赖变更(最多 5 条) | 两个 agent | **自动**,决策发生即写 |
| `TASKS.md` | 进行中 + 最近完成的任务(完成区最多 5 条) | 两个 agent | **手动**,用户说 "update memory" 才写 |
| `LEARNINGS.md` | 值得保留的经验教训(最多 5 条) | 两个 agent | **手动** |
| `SYNC_LOG.md` | 每日交接摘要(最多 7 个日期块) | 两个 agent | **每个写入 session 伴随更新当天块**,这是互相看见对方的窗口 |
| `archive/` | 溢出的旧条目、旧日志 | 两个 agent | 超上限时 |

**Long-Term Memory 区(PROJECT.md 内)是整个系统的"压缩核心"**:其他文件是明细或交接记录,这里是蒸馏后的"项目当前状态",上限约 30 行,要求**重写而非追加**——它永远代表现在,而不是历史。历史去 archive/ 或 git 找。

**模板的组织**:skill 的 `templates/` 分两组——`templates/project/`(协议、两个启动文件、lint 脚本,init 时复制到项目根)和 `templates/memory/`(6 个记忆文件,复制为项目的 `AI_MEMORY/`)。项目里的一切都由 init 铺设,skill 仓库本身不是一个"项目"。

## 五、启动流程:分层读取(token 优化的关键)

最初的设计是"启动读全部 6 个文件",后来改成分层,因为每个 session 都全量读太浪费:

**必读(每次启动,约几百 token)**:

1. `CONFIG.md` —— 知道用什么语言写记忆、git 开没开;
2. `PROJECT.md` 的 Long-Term Memory 区 —— 项目当前状态的最稠密表达(首个 session 读全文);
3. `SYNC_LOG.md` 最近 1-2 个日期块—— **看对方 agent 最近改了什么**,这一步是"伪共享"成立的关键。

**按需读(干相关的活之前才读,而不是启动时)**:

- 要做架构/依赖相关工作 → 先读 `DECISIONS.md`;
- 要开始/继续任务 → 先读 `TASKS.md`;
- 调试、重访踩过坑的区域 → 先读 `LEARNINGS.md`。

这套分层借鉴了 MemGPT/Letta 的内存分级思想:**core memory(常驻上下文,极小)→ recall(按需取)→ archival(冷存储)**。我们的对应物是:Long-Term 区 = core,三个明细文件 = recall,archive/ = archival。

## 六、写入机制:什么时候记忆会被更新(完整触发条件)

### A. 自动写入(agent 不需要任何人开口,发生即写)

1. **做出或更改架构/技术决策** → `DECISIONS.md` 加一条;
2. **依赖变更**(新增/删除/升级任何包)→ `DECISIONS.md` 加一条;
3. **项目目标、结构、约定发生变化** → 更新 `PROJECT.md`;
4. **完成一个任务** → 更新当天 `SYNC_LOG.md` 日期块(注意:**不**自动动 `TASKS.md`,完整任务状态更新仍是手动的;当天交接摘要是防止"Codex 做完了 Claude 不知道、重复劳动"的低成本保险);
5. **首次初始化** → init 完成后在当天日期块写入 `[init] memory initialized`。

### B. 伴随写入(跟随上面任何一次写发生)

6. **每个记忆写入 session**都必须同时更新当天 `SYNC_LOG.md` 日期块,每个被改的文件一条 bullet:`- [HH:MM] [agent] [文件名] 摘要`。同一天只保留一个日期块,避免流水噪音。

### C. 手动写入(只在用户明确要求时:调 share-memory skill 或说 "update memory")

7. **任务进度** → 更新 `TASKS.md`(勾掉完成项、加新任务);
8. **经验教训** → `LEARNINGS.md`(且只在真有值得记的教训时,不凑数)。

### D. 维护性写入(条件触发)

9. **溢出处理**:任何文件超过 5 条时,最老的条目要么(问过用户后)蒸馏进 Long-Term Memory,要么移到 `archive/`;
10. **consolidate**(用户说 "consolidate memory"):合并重复条目、归档被推翻的决策、重写 Long-Term 区、裁剪 SYNC_LOG 到最近 7 个日期块。

### 写入的格式纪律(每一条都适用)

- 条目签名:`### [YYYY-MM-DD HH:MM] [Claude|Codex] 标题`;
- `SYNC_LOG.md` 格式:`## YYYY-MM-DD` + 若干 `- [HH:MM] [Claude|Codex] [filename] summary`;
- `SYNC_LOG.md` 如果当天块不存在,追加到文件末尾;如果已存在,只在写锁内编辑当天块,可追加或压缩 bullet;过去日期块视为关闭,不再编辑,只整块归档;
- **时间戳必须来自系统**(`date "+%Y-%m-%d %H:%M"`),禁止 agent 凭感觉编——LLM 经常不知道今天几号;
- **电报体,每条 ≤3 行**:记忆是查找表不是日记,长篇大论既烧 token 又难检索;
- **写前查重**:同一决策不重复记;推翻旧决策时新条目标 `supersedes [日期]`,而不是留两条互相矛盾的记录;
- **绝不写入 secrets**:API key、密码、token、私有 URL 一律禁止(lint 脚本会扫)。

### 写入准入标准:只记会改变未来行为的事实

ShareMemory 最怕变成"AI 日记"。所以协议规定硬规则:一条内容只有在会改变未来 agent 的行为时才值得进入长期记忆。

**必须写**:已接受的架构/依赖决策;项目目标、范围、工作流、约定变化;能避免重复劳动的任务完成摘要;已验证的 bug 原因和修复;项目相关的用户偏好或硬约束。

**不要写**:原始推理过程、chain-of-thought、未经验证的猜测;没有复用价值的临时错误;代码里一眼能看见的实现细节;大段日志/diff/聊天记录;任何 secret 或不必要的个人数据。

判断句:如果下一个 agent 读到它不会改变做法,就不要写进 durable memory;最多放进当天 `SYNC_LOG.md` 的短 bullet。

## 七、体积控制:为什么记忆不会无限膨胀

膨胀是这类系统最常见的死法——记忆越长,每次启动越贵,最后没人读得完。三道防线:

1. **硬上限**:DECISIONS / LEARNINGS / TASKS(完成区)各 5 条;SYNC_LOG 最近 7 个日期块;Long-Term 区约 30 行。`check_memory.sh` 负责盯着。
2. **渐进式摘要(progressive summarization)**:溢出不是删除,而是"降低保真度"——完整条目 → 蒸馏成 Long-Term 里的一两句 → 原文进 archive/ 冷存储。重要信号保留,细节可追溯。
3. **定期 consolidate**:对抗碎片化,把重复、过期、零散的条目压成稠密状态。

这样整个常驻记忆的体积有数学上限:无论项目跑多久,启动必读内容始终在几百 token 量级。

## 八、冲突、并发与安全

**用户指令 vs 记忆冲突**:用户永远优先,但 agent 必须先指出"这和记忆里 X 月 X 日的决策矛盾"并得到确认,然后更新记忆。两个禁止:不许默默无视用户,也不许默默推翻记忆。(最初设计是"记忆优先",后来反转——人必须是最终裁决者。)

**并发写(已知最大风险)**:两个 agent 同时跑、同时写一个文件仍然可能互相覆盖。协议内置轻量写锁,但它只是防误伤,不是完整并发控制。对策有四层:

- 约定:**不要同时在同一个项目跑两个 agent**(协议明文);
- 锁:写 `AI_MEMORY/` 前用 `mkdir AI_MEMORY/.write.lock` 原子创建锁,写入 owner,失败则停止并读取 owner,不得强行写入;
- 习惯:`SYNC_LOG.md` 当天块可整理,过去日期块不再编辑;摘要文件重写前必须重读当前内容;
- 兜底:**git 历史找回层**——init 时询问用户是否启用(很多用户想要但不知道,所以主动问并解释),启用后每个写入 session 提交一次 `AI_MEMORY/`,任何被覆盖的内容都能从历史恢复。选择记录在 `CONFIG.md` 的 `Git:` 字段;没装 git 或用户拒绝则跳过,功能不受影响。`git init` 必须征得明确同意,绝不静默执行。

**Long-Term 区的特殊风险**:它要求"重写而非追加",重写就可能弄丢对方 agent 写的内容,所以协议强制**重写前必须重新读一遍当前内容**,git 历史做最后兜底。

**软约束问题**:所有规则对 agent 都是"指令"而非"强制",长 session 后期可能遗忘(忘签名、忘日志)。对策是把关键校验做成脚本:`check_memory.sh` 检查条数上限、签名格式、SYNC_LOG 日期块、Long-Term 行数、secrets 泄漏,协议要求每次写入后必须跑。Claude Code 侧未来还可以用 hooks 在 session 开始时自动注入记忆、写入后自动跑 lint;Codex 没有 hooks,这是两边可靠性的已知不对称。

**依赖与安装哲学**:整套系统只依赖 bash + 常见 coreutils(grep/awk/wc/date/sort/uniq),零安装、零联网。git 纯可选。skill **永远不自动安装系统软件**——记忆工具不该动用户的系统,缺什么就告知并降级。

## 九、Token 优化设计汇总

| 手段 | 做法 | 效果 |
|---|---|---|
| 常驻文件极小 | CLAUDE.md / AGENTS.md 各 ~15 行,只指向协议 | 每轮对话的固定开销接近零 |
| skill 按需加载 | init/update/status/consolidate 的细节只在调用 skill 时进上下文 | 平时不占空间 |
| 分层启动 | 必读 3 样(CONFIG + Long-Term + 最近 1-2 个日期块),其余按需 | 启动成本从"全量"降到几百 token |
| 电报体 + 条数上限 | 每条 ≤3 行,每文件 ≤5 条 | 记忆总体积有上限 |
| 渐进式摘要 | 溢出蒸馏进 Long-Term,原文归档 | 信息不丢,常驻体积不涨 |
| 每日摘要而非流水 | SYNC_LOG 只读最近 1-2 个日期块 | 当天任务可长可短,历史不会污染启动上下文 |

## 十、典型工作流演练

**Day 1,Claude Code**:用户打开项目说 "init memory" → skill 问语言(选中文)、问是否启用 git(用户同意)→ 复制模板、采访用户填好 PROJECT.md → SYNC_LOG 创建当天日期块。随后工作中用户决定用 Next.js 而不是 Vite → Claude **自动**往 DECISIONS.md 写一条(签名 + 时间),当天日期块加一条 bullet,跑 lint,git commit。

**Day 2,Codex**:用户打开同一文件夹 → Codex 读 AGENTS.md → 按协议读 CONFIG、Long-Term、SYNC_LOG 最近 1-2 个日期块 → 看到昨天 Claude 选定 Next.js(SSR 需求) → 要做构建相关工作,按需读 DECISIONS.md 全文 → 在 Next.js 前提下干活,装了新依赖 → 自动写 DECISIONS.md + 当天 SYNC_LOG bullet。做完一个任务 → 当天日期块记录任务摘要。

**Day 3,Claude Code**:启动即从日志看到 Codex 昨天的两条变化,无缝接续。用户说 "update memory" → TASKS.md 勾掉完成项;DECISIONS.md 到了 6 条 → 问用户后把最老一条蒸馏进 Long-Term,原文进 archive/。

## 十一、已知局限与未来方向

- **不支持真正并行协作**:已有轻量 lockfile 防误写,但没有事务、merge 或冲突自动解决。
- **规则遵从靠自觉**:lint 只能事后抓,不能事前防。Claude Code 侧可上 hooks 强制;Codex 侧暂无等价机制。
- **SYNC_LOG 是每日摘要没有 diff**:看不到完整改动细节,细节靠 git 历史。
- **扩展第三个 agent**:给它一份启动文件(指向协议、声明 AGENT_NAME)即可,协议本身是 agent 无关的。
- **自动 consolidate**:目前靠用户触发,未来可约定"每 N 次写入后 agent 主动建议 consolidate"。
- **更强 lockfile / hooks / CI 校验**(push 到 GitHub 后可加 Action 跑 check_memory.sh)都是低成本增强,留作迭代。

## 十三、发布前最后一轮边界收口

首个发布前的最后一轮审查补齐了五个边界,并形成协议 v1.0:

- **锁龄报告**:遇到他人持锁必须向用户报告锁的创建时间和年龄(~60 分钟以上且无 agent 运行 = stale 候选),删锁仍须用户确认。锁范围扩大到 AI_MEMORY 内任何写操作(含新建/归档文件)。
- **跨午夜**:"今天" = 写入时刻的系统日期,跨 00:00 即开新日期块。
- **Correction 惯例(最重要)**:关闭的日期块写错了也不可回改;在今天的块写 `[correction]` bullet 指向错误条目,当前视图文件同时加 `supersedes` 修正条目。绝不沉默绕过错误记忆——纠错本身就是交接信息。
- **单块 ≤15 bullets**:防止单个高强度日撑爆启动读取,lint 同步检查。
- **版本升级通道**:skill 声明自带协议版本;status 检测项目 CONFIG 版本落后并提议迁移;新增 migrate 操作;check_memory.sh 核对版本。解决了"协议复制进项目后永远钉死"的长期问题。

历次内部迭代的详细记录在本地 docs/changes/(未随仓库发布)。

## 十四、v1.1:boot 层 import 化与幂等安装器

发布 v1.0 后的第一轮架构评审带来四个升级(memory 数据格式不变):

- **`@AGENTS.md` import**:CLAUDE.md 和 AGENTS.md 原本是对称双胞胎,违反协议层自己的 DRY 原则。现在 AGENTS.md 是 agent-neutral 的共享底座(含条件式 agent 注记),CLAUDE.md 只剩一行 `@AGENTS.md` import + Claude 专属注记——这是 Anthropic 官方推荐的模式,共享规则只存在一处。
- **Marker block 幂等安装**:所有 skill 管理的 boot 内容包在 `<!-- SHAREMEMORY:START/END -->` 里,init/repair/migrate 只插入或替换 block、改前备份、永不碰用户自己的内容,重复 init 不再越追加越乱。lint 新增检查:每文件恰好一个 block、CLAUDE.md 必含 import。
- **`repair memory` 操作**:检测并修复漂移(import 丢失、block 重复、文件缺失、stale 锁),让项目从"一次性模板"变成可长期维护的工具。init 也升级为带状态检测(全新/纯 Claude/纯 Codex/双 agent/已初始化)的严谨 installer。
- **Claude auto memory 策略**:Claude Code 默认开启 machine-local 的私有记忆,存在 split-brain 风险(决策进了 Claude 私有记忆,Codex 看不到)。对策写进 CLAUDE.md 注记:跨 agent 事实必须落 AI_MEMORY/;strict mode(`autoMemoryEnabled: false`)仅作为测试用可选项。

评审中被否决的提案:`AI_MEMORY/BOOT.md`(启动顺序已在协议 §2,再加一个文件就有两处声明同一规则,且启动反而多读一个文件);默认下发 `.claude/settings.local.json`(用户本地文件,skill 不应触碰)。
